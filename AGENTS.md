# AGENTS.md

## Commit Rules

- Перед каждым `git commit` обязательно спрашивать подтверждение у пользователя.
- Сообщение коммита писать в стиле:
  - с маленькой буквы;
  - в форме действия;
  - примеры: `добавляет ...`, `правит ...`.
- Сообщение должно кратко описывать, что делает коммит, без воды.

## Pre-Commit Checklist

- Показать пользователю `git status -sb`.
- По возможности запускать проверки перед коммитом:
  - `pio test -e native`
  - `pio run -e lilygo-t-display-s3 -e lilygo-t-qt-pro`
- Если проверки не запускались, явно указать это перед запросом на коммит.

## Documentation Rules

- Публичные функции и типы документировать в заголовочных файлах (`*.h`) в стиле Doxygen-блоков (`/** ... */`) на русском языке.
- Для каждой публичной функции обязательно описывать:
  - что делает функция;
  - где и как вызывать (`setup`, `loop`, периодичность);
  - блокирующая она или нет;
  - единицы измерения параметров/таймаутов (`ms`, `Hz`, `bytes`);
  - побочные эффекты (GPIO, Wi-Fi, NVS, SD, перезагрузка и т.д.);
  - ограничения по платам (`board-specific` поведение).
- Для функций, небезопасных в прерывании, явно писать `не ISR-safe`/`не вызывать из ISR`.
- В `*.cpp` добавлять только краткие комментарии к сложной логике (state machine, нетривиальные ветки, аппаратные обходные решения), не комментировать очевидные строки.
- Для конфигурационных макросов в `config.h` указывать назначение и единицы измерения прямо рядом с определением.

## Version Rules

- При разработке и изменении функциональности повышать версию приложения в `#define APP_VERSION "..."` в `config.h`.
- Уровень изменения версии (`major` или `minor`) определять отдельно по согласованию с пользователем.

## Git Safety

- Не делать destructive-команды без явного запроса пользователя (`reset --hard`, `checkout --`, и т.д.).
- Не изменять/не откатывать чужие изменения без запроса пользователя.
- Делать коммиты тематическими (один логический блок на коммит), когда это возможно.

## Collaboration Notes

- Перед коммитом кратко перечислять, какие файлы входят в коммит.
- Если есть спорные решения, сначала согласовать с пользователем, потом коммитить.
